Data Strategy · Reliability Engineering

Turning quality into a number you can measure

Global Industrial Manufacturer · Channel Commissioning Software, 70+ Countries
How the offline-first telemetry loop worked
Problem
  • Quality was measured by support tickets — a lagging signal, and the platform's use was predominantly offline by design.
  • Platform ran in 70+ countries, serving 2,500+ enterprise customers.
  • No real baseline existed — quality improvements could be claimed, but not measured.
Impact
  • 165% YoY revenue growth, past €1M ARR in year one.
  • 20% MAU growth.
Approach

The platform was built for a predominantly offline, industrial use case — which made "measure everything" a harder problem than it sounds. Engineers flagged that a real telemetry solution wasn't a sprint-sized fix, so the idea sat benched.

I chose to run the marathon instead of waiting for a sprint-sized shortcut. Working with the team, we designed an offline-first architecture: data captured locally the moment it happened, synced to the server the moment the device reconnected.

  • Reliability — the user's probability of completing the task as intended.
  • Performance — the probability of completing it within a stipulated time.
  • Crash reports — user consent to send log files back, building a real dataset to troubleshoot from.

Every use case had to report all three, or it wasn't considered instrumented.

Principle

You can't improve what you can't measure.
